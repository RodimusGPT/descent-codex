# The transformer

At this level of the descent, the model is not yet a server or a GPU. It is a
function that repeatedly turns context into one next <Token text="token" />. The
same loop will matter later for KV cache and hardware, but first we need the
model-side pieces: tokens, vectors, layers, attention, feed-forward compute, and
sampling.

## Tokenization

The prompt starts as plain text. Before the model can use it, a tokenizer splits
that text into subword pieces and maps each piece to an integer ID from the
vocabulary. Many common words are one token; rare or compound words are assembled
from smaller pieces. The important thing is that the model never sees raw
characters. It sees a sequence of token IDs. In the running prompt,
`The GPU keeps KV cache ready for`, tokenization is the step that turns words and
punctuation into the integer sequence the model can actually process.

<Tokenizer client:visible />

<DeeperBlock summary="Tokenizer detail">
  Production tokenizers are usually BPE-style or unigram-style systems learned from
  text. They keep frequent chunks intact and fall back to smaller chunks for rare
  strings. The result is compact enough for efficient inference while still being
  flexible enough to represent new words.
</DeeperBlock>

## Embeddings

Each token ID indexes a row in an embedding table. That row is a vector: a list of
numbers that places the token in the model's learned space. Tokens that behave
similarly often end up near each other, and useful directions in the space encode
relationships. This is why a token can carry more than spelling; it arrives with
contextual affordances the stack can work with.

<EmbeddingSpace client:visible />

## Position and the causal mask

Embeddings tell the model what each token is, but not where it is. A transformer
also needs position information so `cache ready for decode` is different from
`decode ready for cache`. Modern LLMs commonly use RoPE, or rotary positional
embedding, which rotates query and key vectors in a position-dependent way before
attention scores are computed.

During generation, attention is also causal. The token at position 5 can read
positions 0 through 5, but it cannot read a future token that has not been
generated yet. That mask is what makes the training-time prediction task match
the inference-time loop.

<DeeperBlock summary="Why RoPE matters later">
  RoPE makes relative position visible inside the `Q dot K` attention score. It is
  one reason context length, long-context extrapolation, and KV-cache reuse are
  not just storage questions. The cached keys already contain the positional
  treatment they had when they were created.
</DeeperBlock>

## The stack

The embedded tokens flow through many nearly identical layers. Modern inference
models typically use a pre-norm structure: RMSNorm prepares the stream, attention
moves information between tokens, a residual path keeps the original stream
available, and a feed-forward block does per-token compute. Repeating that block
lets shallow pattern matches turn into more abstract features.

<TransformerStack client:visible />

<DeeperBlock summary="Layer detail">
  A simplified layer is `x = x + Attention(RMSNorm(x))`, then
  `x = x + FFN(RMSNorm(x))`. The residual additions are why later layers can refine
  earlier information instead of replacing it wholesale.
</DeeperBlock>

## Attention

Attention is how one token reads other tokens in the same context. For a selected
query token, the model assigns weights over prior and nearby tokens, then blends
their value vectors. Different heads can specialize: one may look backward, one
may bridge syntax, and another may spread context broadly. For the running prompt,
the position after `for` might attend strongly to `GPU`, `KV`, and `cache`
because those earlier tokens constrain what a plausible next token can be.

<AttentionFan client:visible />

## Q, K, V, and multi-head attention

Inside attention, the stream is projected into Query, Key, and Value vectors. A
query asks what the current token needs; keys advertise what other tokens contain;
values are the information that gets mixed back in. Scores come from scaled
`Q dot K`, then softmax turns scores into weights. Multi-head attention runs
several versions of this in parallel. GQA and MQA share K/V heads across query
heads, which matters later because fewer KV heads shrink the cache in Part 3.

<QKVMultiHead client:visible />

<DeeperBlock summary="Attention math">
  For one head, the core operation is
  `softmax(QK^T / sqrt(d_head)) V`. Causal masking prevents a token from reading
  future tokens during generation.
</DeeperBlock>

<DeeperBlock summary="Why GQA and MQA exist">
  Full multi-head attention stores separate keys and values for every query head.
  Grouped-query attention shares each K/V head across a group of query heads, and
  multi-query attention shares one K/V set even more aggressively. The benefit is
  a smaller KV cache and less memory traffic during decode. The cost is less K/V
  specialization, so model designers choose the sharing pattern rather than
  treating it as a free compression step.
</DeeperBlock>

## FFN and MoE

After attention moves context around, the feed-forward block does most of the
per-token compute. Dense models run the same FFN for every token. Mixture-of-
Experts models replace that with many expert FFNs and a router that activates
only a top-k subset. That is how a model can have many total parameters while
running only a smaller active fraction for a given token.

<MoERouter client:visible />

<DeeperBlock summary="Active vs total parameters">
  In an MoE layer, the router scores experts and sends the token to only a few of
  them. A model might store more than 100B total parameters while activating only
  a few billion per token. The inactive experts still consume memory, but they do
  not all run on every token.
</DeeperBlock>

## Logits to token

At the end of the stack, a projection maps the final hidden state to one logit per
vocabulary item. Temperature changes how sharp the distribution is. Top-k keeps
only the highest ranked candidates; top-p keeps the smallest set whose cumulative
probability crosses a threshold. A token is sampled, appended to the context, and
the whole loop repeats. That autoregressive loop is what makes the KV cache in
Part 3 necessary. If the next sampled token is <Token text="decode" />, the next
step runs again with `The GPU keeps KV cache ready for decode` as the longer
context.
